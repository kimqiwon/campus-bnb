## 1. 沟通机制

### 1.1 沟通渠道
| 渠道 | 用途 | 频率 |
|-----|------|------|
| 腾讯会议/屏幕共享 | 结对编程实时协作 | 每次Pair Session |
| 微信群 | 日常非紧急沟通 | 随时 |
| GitHub Issues | 任务跟踪与Bug记录 | 每日 |
| GitHub PR Comments | 代码审查反馈 | 每次PR |

### 1.2 每日站会（5-10分钟）
- 时间：每天上午 9:00
- 内容：昨日进展 + 今日计划 + 遇到的阻碍

### 1.3 沟通原则
- **遇到分歧时**：先各自阐述理由5分钟，若无法达成一致，由**领航员**做最终决策，事后在复盘会议中讨论
- **代码风格冲突**：统一使用 ESLint + Prettier 配置，以工具为准
- **技术选型分歧**：优先选择两人都熟悉的方案，而非最"先进"的方案

## 2. 冲突解决方式

| 冲突等级 | 解决方式 | 预计耗时 |
|---------|---------|---------|
| 语法/命名差异 | 遵循 ESLint 规则，直接修复 | 1-2分钟 |
| 实现方案差异 | 领航员决策 + 记录到决策日志 | 5-10分钟 |
| 架构设计分歧 | 各自画出方案草图，对比讨论后投票 | 15-20分钟 |
| 无法调和的分歧 | 咨询老师/助教 | 课后 |

## 3. 版本控制规范

### 3.1 Git Flow 工作流
```
main          ← 生产分支（只接受 develop 的 PR 合并）
  └─ develop  ← 开发主分支
       ├─ feature/tenant-module   ← 租客模块
       ├─ feature/room-module     ← 房源模块
       ├─ feature/booking-module  ← 预订模块
       └─ feature/settlement-module ← 结算模块
```

### 3.2 分支命名规范
- 功能分支：`feature/<模块名>-<简述>`
- 修复分支：`fix/<问题简述>`
- 文档分支：`docs/<文档名>`

### 3.3 Commit Message 规范
```
<type>(<scope>): <subject>

类型（type）：
  feat     - 新功能
  fix      - 修复Bug
  docs     - 文档变更
  style    - 代码格式（不影响功能）
  refactor - 重构
  test     - 测试相关
  chore    - 构建/工具变更

示例：
  feat(tenant): 实现租客注册接口
  fix(booking): 修复日期冲突校验逻辑
  docs: 更新结对编程计划
```

### 3.4 PR 规范（★核心要求★）
1. **严禁直接推送到 `develop` 或 `main` 分支**
2. 每个功能开发在 `feature/*` 分支上进行
3. 完成后发起 PR 到 `develop` 分支
4. PR 标题格式：`[模块名] 功能简述`
5. PR 描述必须包含：
   - 实现的功能概述
   - 测试截图（如有前端改动）
   - 关联的 Issue 编号
6. **PR 必须经另一人 Review 并 Approve 后才能合并**
7. Review 必须留下实质性评论（不能只写 LGTM）
8. 合并后删除功能分支

### 3.5 禁止事项
- ❌ 同一个分支两人同时推送
- ❌ 直接在 main/develop 上提交
- ❌ 未经 Review 直接合并 PR
- ❌ 凌晨集中提交代码（说明是分头写然后合，非真正结对）
- ❌ 一个人在 PR 下只写 "LGTM" 就 Approve
